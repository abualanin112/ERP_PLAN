# Business Requirements & Logic Workspace

## Purpose

This folder prepares the business side of the PropTech ERP/CRM project before converting it into UX flows, screens, and detailed backend implementation.

Use these files as the source for:

- Business requirements.
- Business rules.
- Domain workflows.
- User roles and responsibilities.
- State/status transitions.
- Future UX journey mapping.
- Future API and database planning.

## File Map

1. [01-business-context.md](01-business-context.md)  
   Defines product vision, target users, business goals, and MVP scope.

2. [02-actors-and-permissions.md](02-actors-and-permissions.md)  
   Defines user types, responsibilities, authority boundaries, and permission assumptions.

3. [03-core-domains.md](03-core-domains.md)  
   Defines main business domains such as customers, leads, properties, reservations, contracts, payments, documents, and reporting.

4. [04-business-workflows.md](04-business-workflows.md)  
   Defines end-to-end business workflows that can later become UX journeys.

5. [05-business-rules.md](05-business-rules.md)  
   Defines rules, validations, policies, and status transitions.

6. [06-data-glossary.md](06-data-glossary.md)  
   Defines business terms and canonical entity meanings.

7. [07-ux-translation-brief.md](07-ux-translation-brief.md)  
   Prepares the handoff from business logic to UX design without designing UI yet.

8. [08-open-questions.md](08-open-questions.md)  
   Captures decisions that must be answered before final UX/backend design.

9. [09-domain-specific-requirements.md](09-domain-specific-requirements.md)  
   Captures the real rental/property business requirements: dynamic pricing, owners, units, contracts, daily accounting, utilities, workers, assets, annual rent, and Paymob.

## Working Rule

These files describe what the business needs and how workflows behave. They should not contain UI layouts, colors, component design, or frontend implementation details.
